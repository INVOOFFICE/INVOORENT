3. SÉCURITÉ CÔTÉ CLIENT                                                12 / 15
--------------------------------------------------------------------------------

  Contexte : pas de serveur, données 100 % locales.

  [OK] PBKDF2-SHA256          210 000 itérations pour le mot de passe login
                              et la clé maître. Standard industrie 2024.

  [OK] AES-GCM OPFS           Données chiffrées au repos. Clé non-exportable
                              (CryptoKey) stockée dans IndexedDB.

  [OK] Brute-force            Lockout 30 s après 5 tentatives, persisté en
                              sessionStorage (login) et localStorage (master).
                              TTL 24 h pour reset automatique.

  [OK] Migration DK           La dérivée brute (dk en clair) supprimée de
                              localStorage après migration vers CryptoKey ;
                              seul un vérificateur (dkv = SHA-256(dk)) reste.

  [OK] Timing-safe login      Comparaison PBKDF2 par XOR caractère à
                              caractère (corrigé — anciennement got===rec.dk).

  [OK] escapeHtml()           Systématique dans tous les renderers dynamiques
                              (renderVehicules, renderClients, renderAlerts,
                              rapport-modals, etc.).

  [~~] LICENSE_SECRET         'INVRENT9338' visible dans license-activation.js.
                              Par design (local-first), mais toute personne
                              inspectant le JS peut générer des licences.

  [~~] CSP unsafe-inline      Nécessaire pour les onclick= dans les templates.
                              Neutralise la protection XSS par nonce/hash.

  [~~] Demo token             btoa(JSON.stringify({ts})) — falsifiable depuis
                              la console navigateur. Acceptable pour une démo
                              locale sans enjeu de sécurité réel.

6. FIABILITÉ DES DONNÉES                                               12 / 15
--------------------------------------------------------------------------------

  [OK] Double-write           save() écrit localStorage de façon synchrone
                              puis OPFS en async. Pas de perte sur fermeture
                              brutale du navigateur.

  [OK] Réconciliation TS      Si localStorage plus récent que OPFS au
                              démarrage (ex. : autre onglet), merge automatique
                              avant de peupler le cache mémoire.

  [OK] Réparation OPFS        _repairOpfsFromLocalStorage() recrée le fichier
                              OPFS depuis localStorage si le fichier est absent
                              ou corrompu.

  [OK] Backup/restore         invooValidateBackupStructure() vérifie la
                              structure avant import. Test Node automatisé
                              (npm test). Import avec confirmation explicite.

  [OK] Conflit de réservation Chevauchement de dates détecté et bloqué à
                              la sauvegarde, avec nom du client en conflit.

  [OK] Validation formulaires Immatriculation (3 formats marocains + étranger),
                              email, téléphone, dates, tarif ≥ 0, année 1950-2100.

  [OK] Race condition corrigée _memCacheVersion empêche _syncCacheFromOPFS()
                              d'écraser un save() arrivé pendant son exécution
                              async (corrigé — 4 avril 2026).

  [OK] Dirty flags Supabase   autoloc:saved dispatché après chaque syncKey()
                              pour mettre à jour les _dirty flags (corrigé).

  [~~] uid() collision        Probabilité très faible (~0,3 % sur 1 000
                              enregistrements simultanés) mais pas de
                              UUID v4 complet (122 bits d'entropie).

  [~~] 3 saves consécutifs    saveReservation() peut déclencher jusqu'à
                              3 save(KEYS.veh) successifs. Non-atomique :
                              3 événements autoloc:saved, 3 re-renders
                              potentiels.

  [~~] Supabase timestamp     Merge last-write-wins basé sur updatedAt.
                              Drift d'horloge entre appareils peut favoriser
                              une version plus ancienne.

================================================================================
  POINTS FORTS
================================================================================

  - Architecture storage exemplaire pour une app locale : OPFS chiffré AES-GCM
    + localStorage mirror + reconciliation + réparation automatique.

  - Authentification sérieuse : PBKDF2-SHA256 / 210 000 itérations,
    brute-force protection, clé AES non-exportable en IndexedDB.

  - Offline réellement fonctionnel : toutes les features disponibles sans
    réseau, sync Supabase optionnelle et non bloquante.

  - Module .attach() cohérent : dependency injection propre entre les ~15
    modules extraits et le core.

  LEVIERS D'AMÉLIORATION (impact / effort)
================================================================================

  PRIORITÉ HAUTE (gain rapide)

  1. Tables responsives
     Ajouter overflow-x: auto sur le wrapper des tableaux + media query
     pour basculer en layout card sur mobile (< 600 px).
     Gain estimé : +4 pts UI/UX.

  2. ARIA sur les modales
     Ajouter role="dialog", aria-modal="true", aria-labelledby sur chaque
     .modal-overlay ; focus automatique sur le premier champ à l'ouverture.
     Gain estimé : +2 pts accessibilité.

  3. aria-label sur les boutons icônes
     Remplacer title="..." par aria-label="..." sur tous les btn-icon
     (Modifier, Supprimer, Historique, etc.).
     Gain estimé : +1 pt accessibilité.

  PRIORITÉ MOYENNE

  4. Focus trap dans les modales
     Intercepter Tab/Shift+Tab pour rester dans la modale ouverte.
     Gain estimé : +1 pt accessibilité (WCAG 2.1 niveau A).

  5. Continuer l'extraction de 01-app-core.js
     Extraire : dashboard + alertes, CRUD véhicules/clients, section auth.
     Gain estimé : +2 pts qualité code.

  6. Consolider les saves dans saveReservation()
     Réduire les 3 save(KEYS.veh) en un seul appel consolidé.
     Gain estimé : +1 pt fiabilité + UX (moins de re-renders).

  PRIORITÉ BASSE

  7. Remplacer uid() par crypto.randomUUID()
     Disponible dans les contextes sécurisés (HTTPS/localhost).
     Gain estimé : élimination du risque de collision.

  8. Audit contrastes WCAG AA
     Vérifier les combinaisons --text3 / --text2 sur les fonds sombres.
     Gain estimé : +1 pt accessibilité.
